软路由高连接数与 PPPoE 无法上网问题笔记

以软路由实际运行 OpenWrt / Kwrt 系统为例
适用架构：Qualcomm IPQ 系列 / 其他 ARM / x86 OpenWrt 路由器
系统类型：OpenWrt / Kwrt
场景说明：大量 TCP/UDP 长连接运行下

问题描述

在软路由中，尤其是在做高并发业务（如 大量设备连接）时，可能会出现以下现象：

  • PPPoE 接口显示已连接，宽带拨号正常。
  • 但是网络无法访问互联网，尤其是 IPv6 网站或双栈网站。
  • 系统连接数 /proc/sys/net/netfilter/nf_conntrack_count 可能已经很高，nf_conntrack_max 为 65535 仍然无法解决问题。

这种现象在流量较大、长连接较多的情况下尤为明显。

活动连接数与流量卸载的关系

软路由在处理大量上行/下行流量时，会产生大量 活动连接（active connections / conntrack entries），主要通过 netfilter conntrack 表 记录。

参数 默认值 说明
nf_conntrack_max 65535 conntrack 最大条目数（理论最大）
nf_conntrack_count 动态 当前表使用量

现象分析

  • 当路由器运行大量 TCP/UDP 长连接时，活动连接数可能轻松突破 1 万条。
  • 硬件流量卸载（HW Offload）模式下，流量直接在网卡硬件处理，CPU 参与少，转发速度高。
  • 问题在于硬件卸载对 conntrack 管理存在限制，conntrack 表中已记录连接数量一旦超过大约 1 万条，某些 NAT/PPPoE 接口状态可能异常：
    • 拨号连接存在，但无法上网
    • 网关无法正常转发
    • CPU 使用率低，但网络不可用

这是因为硬件卸载绕过了软件层部分连接管理逻辑，超大量连接下软件无法及时回收 NAT/PPPoE 相关状态。

软件流量卸载与关闭卸载的差异

流量卸载类型 优点 缺点 使用场景
硬件流量卸载（HW Offload） 转发速度高，CPU 占用低 高连接数容易出现 NAT 异常；conntrack 超过 1 万易导致拨号失效 高速转发、低 CPU 消耗环境
软件流量卸载（SW Offload） CPU 可全程管理连接表，conntrack 管理完整 转发速度受 CPU 限制，性能下降 高连接数管理、低速/稳定性优先
无卸载 全程 CPU 管理 最大 CPU 消耗，性能最低 调试、实验或少量流量场景

结论：在高速转发场景下，必须保持硬件卸载以保证带宽性能，同时通过 优化连接超时和自动重拨策略缓解高连接数引起的异常。

优化方案

方案 1：调整 TCP / UDP 连接超时

建议调整到适中值：

类型 原始值（秒） 建议值（秒） 说明
TCP ESTABLISHED 7440 3600 长连接仍可维持 1 小时
TCP CLOSE_WAIT 60 30 尽快释放资源
TCP FIN_WAIT 120 30 避免僵尸连接
TCP TIME_WAIT 120 30 保持 TCP 协议完整性
UDP 60 30 UDP 短连接快速回收
UDP STREAM 180 60 UDP 长流适度保留

方案 2：自动检测接口并重启 PPPoE 脚本

特点：

  1. 自动检测所有接口，选择可访问公网的 IPv6 或 IPv4 接口。
  2. 排除保留地址和本地链路（如 fe80::/10、fc00::/7、fd00::/8、::1 等）。
  3. 自适应 ping 间隔：网络正常时间隔长，异常时缩短，加快故障响应。
  4. 日志保留 7 天自动清理。
  5. 可随系统启动自启。

总结

  • 活动连接数过高 + 硬件流量卸载可能导致 PPPoE 接口“连上但无法上网”。
  • 软件卸载或关闭卸载可以避免 conntrack 限制，但可能降低性能。
  • 调整 TCP/UDP 超时可释放僵尸连接，减少连接占用压力。
  • 自动检测接口 + 自适应 ping + 日志轮转脚本可保证 PPPoE 长时间稳定。
  • IPv6 检测采用排除法，兼容多运营商公网 IPv6 地址，IPv4 作为备选。
